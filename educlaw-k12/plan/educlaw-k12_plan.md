# EduClaw K-12 Extensions — Implementation Plan

## 1. Product Overview

- **Product name:** educlaw-k12
- **Display name:** EduClaw K-12 Extensions
- **Description:** K-12 specific features including discipline management, student health records, special education (IDEA/IEP), and grade promotion. Sub-vertical of the parent EduClaw SIS product.
- **Parent product:** educlaw (112 actions, 32 tables, 8 domain modules)
- **Parent path:** `educlaw`
- **Architecture:** Sub-vertical — standalone skill package with FK references to parent tables; does NOT modify parent tables (one schema migration exception noted below)
- **Database:** Same SQLite database as parent (`~/.openclaw/erpclaw/data.sqlite`)

### Target Domains and Scope

| Domain | Scope |
| --- | --- |
| **discipline** | Behavioral incident recording, consequence assignment, guardian notification, IDEA MDR workflow, PBIS positive behavior recognition, state discipline reporting |
| **health_records** | Student health profiles, nurse office visits, medication catalog and administration logging, immunization records and waiver management, immunization compliance auditing |
| **special_education** | IDEA Part B full pipeline: referral → evaluation → eligibility → IEP → goals → services → progress monitoring; Section 504 accommodation plans; MDR linkage to discipline |
| **grade_promotion** | End-of-year promotion/retention review, multi-stakeholder decision records, parent notification, batch grade-level advancement, at-risk early warning and intervention plans |

### Foundation Dependencies

| Foundation Skill | Purpose |
| --- | --- |
| erpclaw-setup | `company` table (company_id FK on all K-12 tables), `erpclaw_lib` shared library (db, audit, naming, response, validation) |
| erpclaw-gl | Not directly used by K-12 extensions (no fee/accounting actions in scope) |
| erpclaw-selling | Not directly used (billing handled by parent educlaw) |
| erpclaw-payments | Not directly used (payments handled by parent educlaw) |
| erpclaw-hr | `employee` table — used as IEP team member reference for non-teacher staff (counselors, psychologists) |
| **educlaw (parent)** | `educlaw_student`, `educlaw_guardian`, `educlaw_student_guardian`, `educlaw_academic_year`, `educlaw_academic_term`, `educlaw_instructor`, `educlaw_data_access_log`, `educlaw_notification`, `educlaw_consent_record` |

### Schema Migration Coordination Required

The parent `educlaw_data_access_log.data_category` CHECK constraint covers:
`'demographics','grades','attendance','financial','health','discipline','communications'`

K-12 special education requires logging under `'special_education'`. **File an issue with the educlaw maintainers** to add `'special_education'` to the CHECK constraint before deploying this sub-vertical.

## 2. Domain Organization

| Domain | Module File | Scope |
| --- | --- | --- |
| discipline | scripts/discipline.py | Incident management, per-student consequences, MDR workflow, PBIS recognition, guardian notifications, discipline reports |
| health_records | scripts/health_records.py | Health profiles, nurse visits, medication catalog and administration log, immunization records, waiver management, compliance auditing |
| special_education | scripts/special_education.py | IDEA referral-to-IEP pipeline, goal/service/progress tracking, IEP team, service compliance, Section 504 plans |
| grade_promotion | scripts/grade_promotion.py | Promotion reviews, decisions, batch grade advancement, at-risk early warning, intervention plans |

**Router:** `scripts/db_query.py` — imports ACTIONS from all four domain modules and merges into single router. Follows identical pattern to parent educlaw router.

## 3. Database Schema

### 3.1 Tables (New) — 23 Tables Total

All tables:
- Use `id TEXT PRIMARY KEY` (UUID4, generated by Python `uuid.uuid4()`)
- Use `TEXT` for all monetary/decimal values (Python Decimal compatible)
- Use `TEXT NOT NULL DEFAULT ''` for date/timestamp fields (ISO 8601)
- Use `INTEGER NOT NULL DEFAULT 0` for boolean fields
- Use `TEXT NOT NULL DEFAULT '[]'` for JSON array fields
- Use `TEXT NOT NULL DEFAULT '{}'` for JSON object fields
- Prefix: `educlaw_k12_`

## 5. Workflows

### 5.1 Discipline Incident Workflow

```
Trigger: Staff observes/receives report of student behavior incident
→ add-discipline-incident (header: date, time, location, type, severity)
→ add-discipline-student (one per student: role, IDEA status)
→ add-discipline-action (one per consequence per student: type, dates, duration)
  └─ IF suspension: get-cumulative-suspension-days
       └─ IF IDEA-eligible AND cumulative days ≥ 8: ALERT admin
       └─ IF IDEA-eligible AND cumulative days ≥ 10: mdr_required=1, NOTIFY sped coordinator
→ notify-guardians-discipline (for each offender/victim)
→ close-discipline-incident (admin review complete)

MDR Branch (IDEA students ≥10 days):
→ add-manifestation-review (MDR meeting within 10 school days)
→ update-manifestation-review (record Q1, Q2 results, determination)
  └─ IF manifestation: return to placement; activate FBA/BIP
  └─ IF not manifestation: continue removal; maintain FAPE services
```

**GL/SLE implications:** None. Discipline has no financial posting.

---

### 5.2 Health Records Enrollment Workflow

```
Trigger: New student enrollment OR annual health form renewal
→ add-health-profile (allergies, conditions, physician, insurance)
→ add-immunization (one per dose per vaccine: CVX code, lot, date, provider)
  └─ check-immunization-compliance (compare against grade-level rules)
       └─ IF non-compliant: update-health-profile (set provisional flags)
       └─ IF waiver: add-immunization-waiver
→ add-student-medication (for each authorized school medication)
→ verify-health-profile (nurse sign-off: status → active)
```

---

### 5.3 Nurse Office Visit Workflow

```
Trigger: Student arrives at health office
→ get-health-profile (FERPA logged; nurse reviews allergies/conditions)
→ add-office-visit (immutable: complaint, vitals, assessment, treatment, disposition)
  └─ IF medication administered: log-medication-admin (decrement supply)
  └─ IF sent home: parent_contacted=1, record contact time and response
  └─ IF emergency: get-emergency-health-info (FERPA emergency exception log)
```

### 5.4 Special Education Full Pipeline Workflow

```
Trigger: Concern raised about student's learning or behavior
→ create-sped-referral (referral_date, source, reason, prior interventions)
→ update-sped-referral (status → consent_pending; record consent_request_date)
→ update-sped-referral (consent_received_date → auto-calc evaluation_deadline +60 days)
  └─ [SYSTEM ALERT: flag at 45 days remaining in evaluation window]
→ add-sped-evaluation (one per assessment type: psych, academic, speech, OT, PT, etc.)
→ update-sped-referral (status → evaluation_complete)
→ record-sped-eligibility (determination: eligible/ineligible; disability categories; iep_deadline auto-calc +30 days)
  └─ [SYSTEM ALERT: flag at 20 days remaining for IEP deadline]
  └─ IF ineligible → consider add-504-plan

IEP Development:
→ add-iep (draft; PLAAFP, LRE, assessment participation, transition if ≥16)
→ add-iep-goal (one per measurable annual goal: baseline, target, method, frequency)
→ add-iep-service (one per mandated service: type, setting, minutes/week, provider)
→ add-iep-team-member (parent, general ed teacher, special ed teacher, admin, others)
→ activate-iep (parent signature received → status active; prior IEP → superseded)

Ongoing Service Delivery:
→ log-iep-service-session (each session: date, minutes, notes)
→ get-service-compliance-report (planned vs. actual check; alert on gaps)

Progress Monitoring (each reporting period):
→ record-iep-progress (per goal: rating, evidence, parent notes)
→ generate-iep-progress-report (parent-facing progress report)

Annual Review (at annual_review_due_date):
  [SYSTEM ALERT: 30 days before due]
→ add-iep (new version; increment iep_version; parent_iep_id → prior)
→ activate-iep (prior → superseded)

Triennial Re-evaluation (every 3 years):
  [SYSTEM ALERT: 90 days before due]
→ create-sped-referral (new referral for re-eval)
→ [repeat evaluation pipeline]
```

**GL/SLE implications:** None in v1. (Medicaid billing for services deferred to v2.)

### 5.6 Grade Promotion End-of-Year Workflow


### 7.3 Invariants

**Discipline:**
1. A `closed` discipline incident cannot be re-opened or modified (status is terminal for most fields)
2. `cumulative_suspension_days_ytd` is always the sum of suspension duration_days for a student within a single academic year
3. An MDR is required before processing expulsion for any IDEA-eligible student

**Health Records:**
1. One and only one health profile per student (`UNIQUE` on `student_id`)
2. `educlaw_k12_health_visit`, `educlaw_k12_medication_log`, and `educlaw_k12_immunization` records are immutable after creation
3. Immunization compliance checks must respect grade-level-specific requirements, not school-level generic requirements
4. Medical waivers without a physician name are rejected

**Special Education:**
1. At most one IEP with `iep_status='active'` per student at any time
2. IEP goals and services are immutable — changes require new IEP version
3. `evaluation_deadline` must always equal `consent_received_date + 60 calendar days`
4. `iep_deadline` must always equal `eligibility_meeting_date + 30 calendar days`
5. If `is_eligible=0`, `disability_categories` must be empty; `primary_disability='not_applicable'`
6. `transition_plan_required=1` is mandatory if student date_of_birth indicates age ≥ 16 at IEP start_date

**Grade Promotion:**
1. One promotion review per student per academic year (UNIQUE constraint)
2. One promotion decision per review (UNIQUE on `promotion_review_id`)
3. `educlaw_k12_promotion_decision` is immutable after creation
4. `batch-promote-grade` is idempotent — checks current `grade_level` before incrementing
5. `submit-promotion-decision` is blocked if any sibling appeal is still `is_appealed=1` without an `appeal_outcome`
